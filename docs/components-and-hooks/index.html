Overview

This section lists the React components that are available to the MDX files in the src/docs directory (or the docsPath defined in the theme options) of your Chicago Docs site. It also lists the available hooks that can be used in custom components to fetch data about your docs.

All components described on this page are available automatically to all MDX files. There is no need to import them into your MDX files before using them.

Chicago Docs Components

Chicago Docs comes with several components that help make it quick and easy for your users to find information on your product or API. Features such as site search and sidebar navigation mean your users won't get frustrated trying to find what they need.

Terminology

In this section, a child is a block of Markdown text between JSX tags (shortcodes); these blocks of text are passed to the React component as a prop called children. There should be a blank line above and below each Markdown child.

A prop is a value passed to the React component directly.

In the following example, the text "Attention!" is a child of the Alert component, and variant is a prop (the string "warning" will be passed to the component to determine the background color of the Alert box).

See also Guide to MDX: JSX Syntax.

example.mdx

MDX
---
title: Title of example MDX file
---
## Example Subheader
<Alert variant="warning">
Please pay attention to me!
</Alert>

Alert

The Alert component creates a colored text box with a message. Try it out in the live example below.

Consider this a warning!
<Alert variant="warning">
Consider this a warning!
</Alert>

Alert props

  • variant

    string
    optionaldefault: "information"

    Style variant, one of "information" (blue), "success" (green), "warning" (orange), and "error" (red).

  • children

    React node or array of nodes
    required

    MDX text placed between Alert tags. This text is rendered inside the Alert box.

The Theme-UI style variants for this component are listed in the alerts object.

The following variants are available:

variant: "information"

variant: "success"

variant: "warning"

variant: "error"

CodeExample

The CodeExample component is used to display one or more code examples in a container. It was designed with API docs in mind. Create language-specific examples for calling an API method, and display them all in one place! Only one code example is displayed at a time; the user chooses an example from a select box at the top of the CodeExample component.

A copy button is also included.

This meta-example demonstrates the CodeExample component using the CodeExample component:

example.mdx

MDX
<CodeExample title="GET" labels={['JavaScript', 'curl']}>
```js
const response = await fetch('https://hipsum.co/api/?type=hipster-centric&sentences=3');
const hipsterIpsum = response.json();
```
```sh
curl -s 'https://hipsum.co/api/?type=hipster-centric&sentences=3'
```
</CodeExample>

This results in the following component:

GET

const response = await fetch('https://hipsum.co/api/?type=hipster-centric&sentences=3');

CodeExample props

  • title

    string
    optionaldefault: '' (empty string)

    Title appearing in the top bar of the CodeExample container.

  • labels

    array of strings
    optionaldefault: []

    Array of strings describing the language or other aspect of each code block (the number of labels should equal the number of code blocks and be in the same order). If more than one label is present, the labels are displayed in a select box allowing users to switch between examples. A single label will be displayed as text.

  • children

    React node or array of nodes
    required

    MDX code blocks placed between CodeExample tags.

This component is meant to be used in MDX files only, as it expects code blocks produced by gatsby-plugin-mdx.

ExternalLink

This component creates an external link that opens in a new tab, which is indicated by an external-window icon. This component is used automatically in docs for any Markdown or HTML links with an href attribute starting with 'http'. For example, <a href="https://en.wikipedia.org">Wikipedia</a> becomes Wikipedia. You can also use the component directly if you prefer:

example.mdx

MDX
<ExternalLink href="https://en.wikipedia.org/wiki/Cats_and_the_Internet">Cats and the Internet</ExternalLink>

ExternalLink props

  • href

    string
    required

    External URL that will open in a new tab when the component is clicked.

  • children

    React node
    required

    MDX text placed between ExternalLink tags, used to display the link.

Image

The Image component from Chicago Docs uses gatsby-image to optimize and lazy load an image. This component renders a fluid image that adjusts to the size of your screen. You can also use the Img component from gatsby-image directly. See gatsby-image for guidance.

example.mdx

MDX
<Image src="aleksey-milov-unsplash.jpg" alt="Cat loaf" />

This will render a fluid image (try changing the size of your screen):

Photo by Aleksey Milov on Unsplash

Image props

  • src

    string
    required

    Filename of the image asset (must live in your site's designated assets folder, such as src/assets). Do not include the path.

  • alt

    string
    optionaldefault: '' (empty string)

    Text that describes the image.

  • title

    string
    optionaldefault: value of alt prop

    Text that appears in a tooltip when the image is moused over.

PostList

This component is intended to be used in the index.mdx file of a directory. It prints a list of posts in that directory. For example, placing this at your-site.com/tutorials/configuration/ will return a list of all posts with the base path of /tutorials/configuration/, such as /tutorials/configuration/how-to-configure-gatsby.mdx.

The following example demonstrates use of the PostList component:

src/docs/tutorials/index.mdx

---
title: Table of contents

showTOC: false
---
This component will print a list of docs in the `tutorials` directory of the `docsPath` directory, such as `src/docs/tutorials/how-to-install-ubuntu.mdx`.

PropertyTable

This table component provides a neat way of listing the properties or attributes of a React component or other entity. It wraps a regular Markdown table that has specific header values (Name, Type, Info, Default, and Description). In fact, it's used to create its own props table here:

example.mdx

MDX
<PropertyTable title="PropertyTable props">
| Name | Type | Info | Default | Description |
| ---- | ---- | ---- | ------- | ----------- |
| `title` | string | optional | `''` (empty string) | Title that appears in the header of the table. |
| `children` | React node or array of nodes | required | | Markdown table placed between PropertyTable tags. Table headers should be `Name`, `Type`, `Info`, `Default`, and/or `Description` (order not important). Other table headers will be ignored. |
</PropertyTable>

PropertyTable props

  • title

    string
    optionaldefault: '' (empty string)

    Title that appears in the header of the table.

  • children

    React node or array of nodes
    required

    Markdown table placed between PropertyTable tags. Table headers should be Name, Type, Info, Default, and/or Description (order not important). Other table headers will be ignored.

Note. If a value for the Default column is included, it will appear following the string 'default:'. You can get around this by omitting a default value and adding it elsewhere:

example.mdx

MDX
<PropertyTable title="Resource properties">
| Name | Type | Info | Default | Description |
| ---- | ---- | ---- | ------- | ----------- |
| `type` | string | optional (default: `'array'`) | | Type of the resource, one of `'array'` or `'object'`. |
| `owner` | string | required | | Owner of the resource. |
</PropertyTable>

This results in the following table:

Resource properties

  • type

    string
    optional (default: 'array')

    Type of the resource, one of 'array' or 'object'.

  • owner

    string
    required

    Owner of the resource.

TOC

The TOC component is added automatically to MDX files in src/docs (or the docsPath defined in the theme options) when the alwaysShowTOC theme option is set to true. This can be overriden by the showTOC frontmatter field for an individual post.

You can also insert a TOC component in any MDX file. It will produce a TOC for the current file.

example.mdx

MDX
<TOC depth={2} />

This results in the following component (a table of contents for the current doc):

TOC props

  • depth

    number
    optionaldefault: 6

    Depth of nested items to print (between 2 and 6).

  • contents

    object
    optionaldefault: null

    Custom table of contents with the following shape: { items: { title: '', url: '' } }. Used by other components.

  • title

    string
    optionaldefault: '' (empty string)

    Title displayed above list of contents.

  • className

    string
    optionaldefault: '' (empty string)

    Class name added to the outer div tag, which also has a class name of 'toc-container'.

The Sidebar and TOC components both track the active header in the viewport as the user scrolls, and they highlight the corresponding header in their respective list of posts. This highlighting can seem to jump around if you have nested headers in your post. To prevent this, surround the nested sections with an HTML element, such as a <section> or the TwoColumnSection component. Each section should include the nested header and its paragraphs:

## Outer header
<section>
### Nested header
First paragraph.
Second paragraph.
</section>
<section>
### Another nested header
Another paragraph.
</section>

TwoColumnSection

The TwoColumnSection component creates a <section> HTML tag and is styled in a two-column layout where the left column scrolls and the right column is sticky on wide screens. On smaller screens, the second column appears below the first column and is not sticky. The best use for this component is to organize API methods. Describe the method in the left column, and provide request and response examples (or other content) that remain stickied on the right.

TwoColumnSection props

  • children

    React node or array of nodes
    required

    MDX text placed between TwoColumnSection tags. Each child can be any valid MDX text, but it works best with code blocks and CodeExample components.

Use a thematic break (<hr /> or ---) somewhere between the TwoColumnSection tags to separate the content for the two columns (i.e., content before and after this tag will be placed in the left and right columns, respectively).

Internal and External Links

There are three link types with different behaviors on a Chicago Docs site:

TypeBehaviorPathComponentExample
External linkOpens in a new tab and includes a new-window iconStarts with 'http'ExternalLinkGitHub
Internal linkNavigates to other pages on the same site without a page refreshRelative path (e.g., /docs/quick-start/)Gatsby LinkQuick Start
Fragment linkJumps to a different location on the same page without a page refreshFragment path (e.g., #theme-ui-components)<a> tagTheme-UI Components

Create a link using either Markdown or HTML:

[Github](https://www.github.com)
<a href="https://www.github.com" target="_blank" rel="noopener noreferrer">Github</a>

You can also use the Gatsby Link and ExternalLink components directly if you prefer.

Code

Overview

Code blocks can be added to your docs using markdown syntax:

example.mdx

MDX
```js
() => {
  return ...
}
```

The programming language (e.g., js) should be specified after the first three backticks (```) so that Prism.js can determine how to highlight the code. Since prism-react-renderer is used behind the scenes, Chicago Docs supports only those languages supported by that plugin. These languages are listed here.

Line highlighting, line numbers, and live preview are rendered through the metastring, which is the text that follows the three backticks and the language value as described above. For example, in ```js {3}, the metastring is {3} and adds highlighting to line 3 of the code block.

This section describes how to add line highlighting, line numbers, and live previews.

You need to specify the language in order for the metastring value to be used. For example, ``` {3} will not produce line highlighting. If you don't want any Prism code highlighting but want to add line highlighting or other features, you can use the 'shell' language:

```sh {3}

Hooks

This theme's React hooks are a convenient way to access data about your site or the current post. They allow you to access metadata, theme options, sidebar menus, and the table of contents. They're meant to be used in custom or shadowed components. Import them from @kabartolo/gatsby-theme-chicago-docs/src/hooks.

Note that the @kabartolo/gatsby-theme-chicago-docs-core theme also comes with some of these hooks (as noted in the table below).

Use these hooks in your custom components to fetch data:

HookParametersPurposeIn Core
useDocContextFetches the docId and sidebar menu object for the current doc.No
useDocDataFetches all GraphQL data for all docs (see below).Yes
useLocationFetches the browser's location object.No
usePrismThemeFetches the current Prism theme (according to the current color mode).No
useSiteMetadataFetches the site metadata defined in gatsby-config.js.Yes
useTableOfContents{ depth = 6, docId = '', path = '' }Fetches the table of contents for all docs in an array of objects, each object including the doc id, a nested object, and a flatMap array. Pass the docId or current path to get the object for a specific doc. The depth value specifies the depth of the nested object (does not affect flatMap).Yes
useThemeOptionsFetches the Chicago Docs theme options defined in gatsby-config.js.Yes

The useDocData hook returns an array of nodes with the following information about each doc:

id
title
shortTitle
description
path
slug
excerpt
body
showBreadcrumb
showPostNav
showSidebar
showTOC

See Configuration for more details on theme options and frontmatter.

import { Link } from 'gatsby';
import { useDocData } from '@kabartolo/gatsby-theme-chicago-docs/src/hooks';
export default function DocList() {
  const docs = useDocData();
  return (
    <ul>
      {docs.map((doc) => (
        <li key={doc.id}>
          <a href={doc.path}>{doc.title}</a>
        </li>
      ))}
    </ul>
  );
}
